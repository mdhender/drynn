# AGENTS.md — drynn

## Overview

Go web app for competition operations. Echo v5 backend, HTMX/Alpine.js + Go templates frontend, PostgreSQL database. Runtime config loaded from a persisted JSON file with environment overrides.

## Tech Stack

| Layer         | Technology                                                           |
|---------------|----------------------------------------------------------------------|
| Web framework | [Echo v5](https://echo.labstack.com) (`github.com/labstack/echo/v5`) |
| Frontend      | Go `html/template` (`.gohtml`), HTMX, Alpine.js                      |
| Database      | PostgreSQL via `pgx/v5` connection pool (`pgxpool`)                  |
| Migrations    | [Atlas](https://atlasgo.io) (versioned workflow)                     |
| SQL interface | [sqlc](https://sqlc.dev) with `pgx/v5`                               |
| Auth          | JWT (access + refresh) via `golang-jwt/jwt/v5`, bcrypt passwords     |
| IDs           | `github.com/google/uuid`                                             |

## Critical Rules

- **Echo v5, not v4.** Import `github.com/labstack/echo/v5`. Handlers take `*echo.Context` (pointer). Never use `echo/v4`.
- **No `database/sql`.** All DB access via sqlc-generated code backed by `pgx/v5`.
- **No raw SQL in Go.** Queries go in `db/queries/*.sql`, compiled with `sqlc generate`.
- **Never hand-edit `db/sqlc/`.** Generated by sqlc; will be overwritten.
- **Never edit applied migrations.** Create a new migration instead.
- **Never hardcode secrets.** JWT keys, passwords, and database URLs come from config/environment.

## Echo v5 Conventions

```go
// Handler — *echo.Context (pointer, not interface)
func (h *AppHandler) ShowProfile(c *echo.Context) error {
    return c.Render(http.StatusOK, "app/profile", data)
}

// Middleware
func requireRole(role string) echo.MiddlewareFunc {
    return func(next echo.HandlerFunc) echo.HandlerFunc {
        return func(c *echo.Context) error {
            // ...
            return next(c)
        }
    }
}
```

- Routes registered via `e.GET`/`e.POST`; grouped with `e.Group("/app")`.
- Templates rendered with `c.Render(status, "name", data)`. Template names map to layout+page pairs in `internal/server/render.go`.
- Static files served from `web/static/` at `/static/`.

## Handler Architecture

Four handler groups, each a struct receiving service dependencies:

| Handler          | Prefix       | Auth required | Description                        |
| ---------------- | ------------ | ------------- | ---------------------------------- |
| `PublicHandler`  | `/`          | No            | Landing page                       |
| `AuthHandler`    | `/`          | No            | Register, sign-in, sign-out, refresh |
| `AppHandler`     | `/app`       | Yes           | Profile (`/app` → `/app/profile`)  |
| `AdminHandler`   | `/app/admin` | Yes + `admin` | User CRUD                          |

View data structs and helpers live in `handler/view.go`. All view data embeds `BaseViewData` (title, current path, flash, current user).

## Auth

- `internal/auth/` contains `Manager` (token issue/parse/cookies), `KeyStore` (DB-backed signing keys), `RequireAuth` middleware, password hashing, and the `Viewer` type.
- Tokens read from cookies first, then `Authorization: Bearer` header.
- `Viewer` is loaded by the `loadCurrentViewer` middleware and stored in the Echo context. Retrieved via `auth.CurrentViewer(c)`.
- Database roles: `user`, `admin`. Enforced by `requireRole` middleware. Unauthenticated sessions get a synthetic `guest` sentinel viewer from `auth.CurrentViewer` so role checks never need nil guards.
- The server refuses to start without one active signing key per token type.

## Config

Runtime config loaded from a JSON file (default `data/var/drynn/server.json`) with env overrides. Written by `cmd/db init-config`. See `internal/config/config.go`.

Key environment variables: `HOBO_CONFIG_PATH`, `APP_ADDR`, `DATABASE_URL`, `DATA_DIR`, `JWT_ACCESS_TTL`, `JWT_REFRESH_TTL`, `COOKIE_SECURE`. Initial admin users are created with `cmd/db seed-admin`.

## Database & Migrations

Atlas versioned workflow with a named environment:

```bash
# Generate migration after editing db/schema.sql
atlas migrate diff <name> --env local

# Lint latest migration
atlas migrate lint --env local --latest 1

# Apply pending migrations
atlas migrate apply --url "$DATABASE_URL"
```

Edit `db/schema.sql` first, then `atlas migrate diff`. Never modify `atlas.sum` manually.

## sqlc

`sqlc.yaml` maps `db/queries/*.sql` → `db/sqlc/` with `pgx/v5`. Type overrides: `uuid` → `uuid.UUID`, `timestamptz` → `time.Time`.

```sql
-- name: GetUser :one
SELECT * FROM users WHERE id = $1;

-- name: ListUsers :many
SELECT * FROM users ORDER BY created_at DESC;
```

After editing queries: `sqlc generate`.

## Templates

- File extension: `.gohtml`.
- Layouts in `web/templates/layouts/`, pages in `web/templates/pages/`, partials in `web/components/`.
- Each named template is a layout + components + page, executed from the `"base"` block.
- Template names (e.g. `"app/profile"`) are registered in `internal/server/render.go`.
- Custom funcs: `hasRole`, `hasPrefix`, `join`.
- HTMX for server interactions (HTML fragments). Alpine.js for client-side interactivity.

## Code Style

- Standard Go: `gofmt`, `go vet`, short names in tight scopes.
- Handlers are thin; business logic in `internal/service/`.
- Wrap errors: `fmt.Errorf("context: %w", err)`.
- Pass `context.Context` through service and DB calls.
